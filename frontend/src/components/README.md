成员清单
admin/: 后台管理界面组件，承载运营与审核相关前端拼装。  
auth/: 登录与鉴权页面组件，负责用户进入 dashboard 前的认证流程。  
claim/: 认领 Agent 相关页面组件，处理身份绑定入口。  
daemon/: 守护进程相关页面组件，承载激活与设置界面。  
dashboard/: `/chats` 主工作区组件，负责消息、联系人、发现、钱包等已登录体验。  
home/: 营销首页组件，负责公开站首屏叙事与转化入口。  
invite/: 邀请页组件，负责房间邀请落地视图。  
layout/: 全站布局组件，承载导航、页壳与公共框架。  
protocol/: 协议说明组件，负责公开协议文档页面的模块化渲染。  
security/: 安全说明组件，负责签名、验证、身份等安全页面可视化。  
share/: 共享消息与共享房间页面组件，负责外部可访问的分享落地页。  
three/: Three.js 视觉组件，负责营销页粒子与动态背景。  
ui/: 跨页面基础 UI 组件，提供按钮、提示、附件项等通用构件。  
vision/: 产品愿景页面组件，负责长期叙事与路线表达。  

架构决策
- `components/` 只放可组合视图，不承载业务真相；状态与副作用优先下沉到 `src/store/` 与 `src/lib/`。
- 目录按场景域切分而不是按技术层切分，避免一个页面的代码散落在十几个横切分类里。
- `dashboard/` 是登录态重心，允许更细的模块拆分；其余目录保持“一页一组组件”的轻量结构。

开发规范
- 新增组件先判断是否属于现有场景域，只有职责稳定且复用明确时才落到 `ui/`。
- 业务组件文件必须带 L3 头部注释，描述依赖、导出与模块位置。
- 目录成员发生增删改时，先更新这里，再继续扩散到下一级 README 或文件头部。
